T3 Code Audit — Bugs & Improvements

TL;DR

Audited the T3 Code monorepo (server WebSocket handler, web app, client-runtime, contracts) for bugs, maintainability issues, and improvement opportunities. Found 1 critical, 2 high, 4 medium, and 6 low priority findings. The critical finding is a race condition in WebSocket transport disconnect handling. The most impactful fix is the onResubscribe double-firing bug in wsTransport.ts that causes state corruption on reconnect.

Critical

C1: intentionalCloseDepth underflow on concurrent closeSession calls

critical packages/client-runtime/src/wsTransport.ts:262-268

Problem: closeSession() increments this.intentionalCloseDepth at line 263 but the decrement runs in a .finally() callback at line 265. If closeSession() is called concurrently (e.g., from rapid reconnects), the depth counter can underflow below zero, causing isCloseIntentional() to return false for intentional closes. 

this.intentionalCloseDepth += 1;
return session.runtime.runPromise(Scope.close(session.clientScope, Exit.void)).finally(() => {
  this.intentionalCloseDepth = Math.max(0, this.intentionalCloseDepth - 1); // underflow possible
  session.runtime.dispose();
});

Why it matters: When isCloseIntentional returns false during an intentional reconnect, the transport may treat a clean close as a connection error, triggering unnecessary warning logs and subscribe resets.

Recommended fix: Use a ReadonlySet of session IDs or an AtomicReference with compare-and-swap to track intentional close state per-session rather than a shared counter.

C2: onResubscribe callback fires twice per resubscribe

critical packages/client-runtime/src/wsTransport.ts:150-152, 167-171

Problem: When a stream connection drops and the retry loop begins, onResubscribe is called from two separate code paths:

  1. Line 150-152: in the onStreamRequestStart listener (fired when a new WS request for the same tag is detected)
  2. Line 167-171: in the retry loop before reconnecting to the new session

Why it matters: Any subscriber hook that resets state (e.g., clearing a cached VCS status, resetting UI loading indicators) will fire twice. This causes flickering state transitions and can trigger double-side-effects like duplicate API calls. 

// Line 150 (in onStreamRequestStart):
options?.onResubscribe?.();

// ...

// Line 168 (in retry loop):
options?.onResubscribe?.(); // Called again immediately

Recommended fix: Remove the call from either location. The retry loop is the correct place since it is the single path that controls reconnection. The onStreamRequestStart listener should only handle side effects that need to happen after the new stream is created.

High

H1: Effect.ignoreCause({ log: false }) silently swallows setup script failures

high apps/server/src/ws.ts:560

Problem: In recordSetupScriptLaunchFailure(), the error from appendSetupScriptActivity is swallowed with Effect.ignoreCause({ log: false }). Every other usage of Effect.ignoreCause in this file uses { log: true }. 

Effect.ignoreCause({ log: false }),

Why it matters: If appendSetupScriptActivity fails (e.g., because the thread was deleted before the activity was appended), the failure is completely invisible — no log, no metric, no error reporting. A thread that was successfully created but fails silently to report setup activity will appear stuck to the user with no diagnostics.

Recommended fix: Change to Effect.ignoreCause({ log: true }) to at minimum log the failure, or restructure to not ignore at all.

H2: subscribe() retry loop has no cancellation mechanism during dispose()

high packages/client-runtime/src/wsTransport.ts:158-212

Problem: The retry loop in subscribe() checks this.disposed and active only at the top of each iteration. If the loop is blocked on await sleep(retryDelayMs) at line 209, calling dispose() will NOT cancel it — the loop will wait for the full delay, then check active and exit. During that window, the subscription continues attempting to reconnect to a disposed transport. 

await sleep(retryDelayMs); // No abort signal, no timeout race

Why it matters: A disposed transport shouldn't continue making WebSocket connections. The delay could be up to 250ms per the default, but during rapid dispose/reconnect cycles this creates a leak of in-flight connection attempts.

Recommended fix: Use an AbortController that is triggered on dispose, and race the sleep against the abort signal: await Promise.race([sleep(retryDelayMs], abortSignal.then(...)]).

Medium

M1: requestStream silently ignores listener errors

medium packages/client-runtime/src/wsTransport.ts:116-120

Problem: When processing stream values, listener errors are silently caught and ignored: 

listener(value); // If this throws, the error is silently swallowed
} catch {
  // Ignore listener errors so the stream can finish cleanly.
}

Why it matters: Buggy listener callbacks (e.g., in the VCS status stream or thread detail stream) will fail silently. The stream "succeeds" but the listener never processed the event. This can mask data loss.

Recommended fix: At minimum log the error. Or, depending on design, fail the stream since a failed listener means the stream's purpose is compromised.

M2: CursorTextGeneration uses bare Effect.ignore

medium apps/server/src/textGeneration/CursorTextGeneration.ts:107

Problem: Bare Effect.ignore (without { log: true }) completely silences the error from runtime.setMode("ask"): 

yield* Effect.ignore(runtime.setMode("ask"));

Why it matters: If setting the mode fails (e.g., the Cursor runtime is in a bad state), the error is invisible. This is a fire-and-forget operation during text generation that could silently fail to switch modes.

Recommended fix: Change to Effect.ignoreCause({ log: true }) or Effect.catchCause with explicit logging.

M3: subscribe() triggers onResubscribe on initial retry before any value received

medium packages/client-runtime/src/wsTransport.ts:166-172

Problem: The retry loop calls onResubscribe when hasReceivedValue is true (line 166), but if the initial stream never delivers a value and fails, hasReceivedValue stays false — so the first retry after failure won't trigger it. However, the onStreamRequestStart listener (line 141-155) will trigger onResubscribe regardless of hasReceivedValue as long as active is true and the tag matches. This creates inconsistent behavior: onResubscribe fires for the onStreamRequestStart path but NOT for the retry loop path.

Why it matters: Inconsistent firing of onResubscribe makes it unreliable for state-reset logic. Some resubscribes will reset state (when onStreamRequestStart fires), others won't.

Recommended fix: Unify the onResubscribe strategy. Either always fire from the retry loop (remove from onStreamRequestStart), or always fire from the listener, but not both with different conditions.

M4: Effect.ignore (bare) used in 3 adapter files swallows scope-close errors

medium apps/server/src/provider/Layers/CodexAdapter.ts:1451,1637
apps/server/src/provider/Layers/CursorAdapter.ts:467,1043
apps/server/src/provider/Layers/GrokAdapter.ts:324,873,963

Problem: Multiple adapter files use bare Effect.ignore(Scope.close(...)) without logging. If a scope close fails (e.g., a child fiber is still running, a resource finalizer throws), the error is invisible. 

yield* Effect.ignore(Scope.close(session.scope, Exit.void));

Why it matters: Failed scope closes indicate leaked resources (open file handles, running processes, dangling WebSocket connections). Silently ignoring these defeats resource leak detection.

Recommended fix: Use Effect.ignoreCause({ log: true }) on all scope-close operations so failures are at minimum logged.

Low

L1: ws.ts is 1578 lines — single file with all RPC handlers

low apps/server/src/ws.ts

Problem: The WebSocket RPC handler file contains all orchestration, server config, VCS, terminal, cloud, auth, and project handlers in one monolithic function (makeWsRpcLayer). Every handler has access to all 40+ local variables. Testing individual handlers in isolation is impossible without instantiating the entire layer.

Why it matters: This is a major maintenance bottleneck. Adding a new RPC method requires understanding the entire 1578-line closure. The in-scope variables list alone is ~40 dependencies.

Recommended fix: Split handlers into domain modules (e.g., ws/rpc/terminal.ts, ws/rpc/vcs.ts, ws/rpc/orchestration.ts) with each module exporting handler factories that accept only their required services.

L2: Static file serving (GET *) has no authentication

low apps/server/src/http.ts:275-359

Problem: The catch-all GET * route serves static files without any authentication. When running in production/desktop mode with Tailscale Serve or port exposure, any network peer could read the web UI's static assets.

Why it matters: While static assets are typically public for SPAs, directory traversal protection exists but no auth means the full client source map and JS bundle could be read. For a development tool this is low severity, but worth documenting.

Recommended fix: Add a config flag to enable auth on static routes for non-loopback connections, or add a notice in the config docs.

L3: cloud.installRelayClient discards the actual stream error

low packages/client-runtime/src/wsRpcClient.ts:340-355

Problem: When the relay client install stream fails, the error is consumed by the stream mechanism and the fallback error message is generic: "Relay client install stream completed without a final status." The actual error from the server is lost. 

if (installed) { return installed; }
throw new Error("Relay client install stream completed without a final status.");

Why it matters: Users see a vague error message instead of the actual installation failure reason (network timeout, auth failure, disk full).

Recommended fix: Track the stream error and include it in the thrown error message.

L4: clearPersistedServerRuntimeState uses fs.remove with force: true

low apps/server/src/serverRuntimeState.ts:56-60

Problem: The cleanup of the server runtime state file on shutdown uses force: true and Effect.ignore({ log: true }). If the file can't be removed (permissions, filesystem error), the failure is invisible.

Why it matters: A stale runtime state file could cause the next server start to read incorrect state (stale PID, wrong port). Not a data loss risk, but could cause confusing startup failures.

Recommended fix: Log at debug level if removal fails, or don't remove at all — the server already handles stale state gracefully via readPersistedServerRuntimeState's use of Effect.option.

L5: getProjects() and getThreads() in store.ts create new arrays on every call

low apps/web/src/store.ts:500-512

Problem: getProjects() and getThreads() iterate through all project/thread IDs and filter empty entries. These functions create new arrays on every call. In a Zustand app, these are likely called in selectors that run on every state change.

Why it matters: Creates unnecessary garbage collection pressure when there are many projects/threads. Could cause React re-renders if used in useStore selectors that don't memoize.

Recommended fix: Memoize results or restructure the store to pre-compute derived arrays in the reducer.

L6: appendSetupScriptActivity uses tuple workaround instead of Effect.all

low apps/server/src/ws.ts:372-394

Problem: The serverCommandId and serverEventId are fetched via Effect.all (line 374-376) but only one is actually used for the activity ID. The serverEventId is used directly in the activity payload, but it's generated by mapping the UUID from randomUUID which itself is derived from crypto.randomUUIDv4. This adds unnecessary cryptographic work for a non-security-sensitive ID. 

const serverEventId = randomUUID.pipe(Effect.map(EventId.make));
// ...
const commandId = yield* serverCommandId("setup-script-activity");
// activityId is from serverEventId but it's just a unique identifier

Why it matters: Minor performance concern. randomUUIDv4 is cryptographically secure but expensive; a simpler monotonic or random ID would suffice for non-security event/activity IDs.

Architecture Observations

Web store uses denormalized Zustand; mobile uses client-runtime managers

The apps/web/src/store.ts:46-50 has a large TODO comment documenting that the web app still stores shell snapshots and thread details in a denormalized Zustand shape while mobile uses createShellSnapshotManager and createThreadDetailManager from @t3tools/client-runtime. The web's Zustand store has grown to 820+ lines of immutable reducer functions that manually manage thread/message/activity/plan/diff indexes. This duplication represents a significant maintenance burden and divergence risk.

Recommended: Migrate web to use the same client-runtime managers. This has been flagged as TODO(CLIENT-RUNTIME MIGRATION) in the code and prevents both sharing new features and fixing bugs in the state management logic.

ws.ts closure scope captures ~40 services for all handlers

The makeWsRpcLayer function in ws.ts:232-1516 creates a single closure with all ~40 service dependencies that every handler captures. This means even a simple handler like serverGetConfig that only uses 5 services still holds references to all 40. This prevents the GC from cleaning up any service even if only a subset is in use, and couples every handler to every dependency.

Recommended: Each handler should capture only its required services via Layer.effect or dependency injection at the handler level, not at the group level.

What Not to Touch

  • Don't refactor ws.ts splitting yet — The file is 1578 lines and needs splitting, but doing so in isolation before the RPC group migration (which is likely planned given the RPC infrastructure) risks conflicts. Wait for the RPC framework to stabilize.
  • Don't add auth to static file serving — The GET * route intentionally lacks auth (it's a local dev tool). Adding it would break the dev workflow where the Vite dev server proxies requests.
  • Don't rewrite the Zustand store — The TODO mentions a client-runtime migration. Let that migration happen naturally rather than investing in the legacy store.
