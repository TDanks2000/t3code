Speed, Reliability & Performance Audit

T3 Code · 2026-06-25 · apps/server, apps/web, packages/client-runtime

TL;DR

Five substantive performance and reliability gaps identified across the server and web layers. The most impactful is the O(N) flat-array read model in the event projector, which causes linear-time thread scans on every dispatched event and becomes measurably slow at scale. The second-highest value change is confirming and enabling SQLite WAL mode, which costs one line and meaningfully improves write throughput. Three medium-priority issues cover unbounded SQL result sets, cache-key scans on session teardown, and redundant Zustand selector work in the web client. No data-loss or security issues were found in this audit scope.

Codebase Understanding

Stack

  • Runtime: Node ≥24 / Bun, pnpm workspaces, vite-plus build tooling
  • Backend (apps/server): Effect-TS 4.0 beta, @effect/sql-sqlite-bun, WebSocket RPC (effect/unstable/rpc), Anthropic Claude SDK, OpenCode SDK
  • Frontend (apps/web): React, Zustand, TanStack Router, @effect/atom-react
  • Client runtime (packages/client-runtime): Effect Atoms driving RPC subscriptions
  • Contracts (packages/contracts): Shared Schema types, RPC group definitions

Architecture

  • Event-sourced orchestration: commands → events persisted to SQLite, applied to an in-memory OrchestrationReadModel via projectEvent(), and fan-out to projection tables.
  • Provider ingestion: a single makeDrainableWorker serializes all provider runtime events through processRuntimeEvent() in ProviderRuntimeIngestion.ts.
  • Per-event Effect.Cache instances buffer streaming assistant text, segment state, and proposed-plan deltas in memory.
  • WebSocket RPC layer in ws.ts exposes orchestration subscriptions to the web client.
  • SQLite schema has 33 migrations; turn costs and tool invocations were added in migration 033.

Relevant Constraints

  • MAX_THREAD_MESSAGES = 2_000 and MAX_THREAD_CHECKPOINTS = 500 cap in-memory per-thread collections.
  • All provider events funnel through one serial worker; concurrency inside event processing is intentionally limited.
  • STRICT_PROVIDER_LIFECYCLE_GUARD is on by default; lifecycle events gate multiple code paths.

Findings by Priority

High F1 — O(N) read-model array scan on every orchestration event

Files: apps/server/src/orchestration/projector.ts:64–70

Problem: The OrchestrationReadModel stores projects and threads as flat Array<T>. updateThread() calls threads.map(t => t.id === id ? patch : t), which is an O(N) scan and a full array copy on every event. findMessageById and findProposedPlanById in ProviderRuntimeIngestion.ts:116–144 also scan linearly through potentially 2,000 messages per event. At 50 threads × 1,000 messages each, a turn.completed event triggers several O(N) passes through a 50-element thread array and up to four passes through a 1,000-element message array.

Why it matters: The drainable worker is serial. Every extra microsecond of per-event work directly increases event-processing latency and can cause observable lag in streaming responses when many threads are active.

Fix: Replace the threads and projects arrays in OrchestrationReadModel with Map<ThreadId, OrchestrationThread> and Map<ProjectId, OrchestrationProject>. updateThread becomes a Map.set instead of .map(). Consumers that need arrays (snapshot serialization, list RPCs) call Array.from(map.values()) at read time. Similarly, index messages inside each thread by id: messagesById: Map<MessageId, OrchestrationMessage>.

Risk: Medium. This is a structural change to the read model. Every projector case that spreads nextBase must be updated. Existing snapshot serialization/deserialization must convert between Map and Array. Add a projector unit test for the Map path before merging.

High F2 — SQLite WAL mode not confirmed enabled

Files: apps/server/src/persistence/Migrations/001_OrchestrationEvents.ts (and sibling migrations), apps/server/src/persistence/NodeSqliteClient.ts

Problem: The 33 migration files do not show a PRAGMA journal_mode=WAL call. SQLite's default journal mode is DELETE (rollback journal). WAL mode allows concurrent reads during a write and dramatically reduces write latency for workloads like this one — frequent small inserts (events, turn costs, tool invocations) interleaved with reads (snapshot queries, RPC subscriptions).

Why it matters: Without WAL, every write acquires an exclusive lock that blocks all concurrent reads, stalling RPC subscription queries during high-throughput turns.

Fix: In NodeSqliteClient.ts (or the first migration), execute PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; immediately after opening the database. PRAGMA synchronous=NORMAL is safe for this workload (non-critical local data) and reduces fsync overhead significantly.

Risk: Low. WAL is the standard recommendation for application SQLite. It requires the WAL file be on the same filesystem as the DB, which is always true here. Safe to ship with no migration needed — it is a connection-level pragma.

Medium F3 — Unbounded SQL result sets in TurnCost list queries

Files: apps/server/src/persistence/Layers/TurnCosts.ts:56–107

Problem: listTurnCostsByThreadId and listTurnCostsByProjectId both execute SELECT ... FROM projection_turn_costs WHERE thread_id = $id ORDER BY created_at ASC with no LIMIT clause. A thread with 500 turns returns 500 full rows. Since these are returned as ReadonlyArray<TurnCostRow> and may be used for in-memory aggregation, they will grow without bound.

Why it matters: The aggregate queries (aggregateByThreadId, aggregateAll, etc.) already perform the aggregation in SQL — so the list queries are only useful for display or export. Returning unbounded rows for large threads is a reliability risk under memory pressure.

Fix: Add a LIMIT parameter to listTurnCostsByThreadId and listTurnCostsByProjectId, defaulting to 200. Update TurnCostRepositoryShape and callers accordingly. Any place needing totals should use the aggregate queries, not the list queries.

Risk: Low. Adds a default limit that callers can override. Verify all call sites to confirm none relies on the full unbounded result.

Medium F4 — clearTurnStateForSession scans all Effect cache keys

Files: apps/server/src/orchestration/Layers/ProviderRuntimeIngestion.ts:1105–1147

Problem: On session.exited, clearTurnStateForSession calls Cache.keys(turnMessageIdsByTurnKey), Cache.keys(assistantSegmentStateByTurnKey), and Cache.keys(bufferedProposedPlanById), then filters by string prefix. With capacities of 10,000–20,000 entries, this is an O(cache-size) scan on every session exit. Sessions can exit frequently (provider restarts, interrupts, stops).

Why it matters: Under high concurrency (many threads across many sessions), this can cause a measurable pause in the drainable worker every time a provider session exits.

Fix: Maintain a Map<ThreadId, Set<string>> (turn keys per thread) updated eagerly whenever entries are added/removed from the caches. On session exit, use this index to delete only the relevant keys. This reduces the cleanup from O(cache-size) to O(active-keys-for-thread).

Risk: Low-medium. The index must be kept in sync with cache mutations. Write a test verifying no key leaks after session cleanup.

Medium F5 — Redundant Zustand selector work in web client selectors

Files: apps/web/src/workspaceOverviewSelectors.ts:1–131, apps/web/src/store.ts:85–93

Problem: selectSidebarThreadsAcrossEnvironments calls Object.values(state.environments).flatMap(env => Object.values(env.sidebarThreadSummaryById)). This selector is called independently in selectOverviewStats, selectAttentionItems, selectActiveWorkThreads, selectRecentThreads, and selectHasThreads. If a component subscribes to all five, it calls flatMap five times per render. selectRecentThreads additionally spreads and sorts the entire thread array on every invocation.

Why it matters: During active AI turns, Zustand state updates are frequent (streaming message deltas). Redundant array flattening and sorting on each update causes unnecessary CPU overhead in the render loop, especially visible with dozens of active threads.

Fix: Use useMemo or memoized selectors (via zustand/shallow or a selector library like reselect) to compute the flattened thread array once per state change. selectRecentThreads should receive a pre-computed thread list rather than calling selectSidebarThreadsAcrossEnvironments internally. Alternatively, move these derived computations into an Effect Atom so they update reactively and cache automatically.

Risk: Low. Pure selector-layer change; no server-side risk. Verify no behavioral regression by running the existing selector tests.

Low F6 — storeSelectors.ts stub always returns null

Files: apps/web/src/storeSelectors.ts:6–11

Problem: createThreadSelectorAcrossEnvironments is implemented as a stub that always returns null, regardless of the thread ID passed. Any component using this selector will never receive thread data. This is a reliability gap for any future consumer of per-thread detail in the web UI.

Why it matters: Not immediately breaking if no component uses it yet, but it is a silent failure surface — a developer adding a feature that calls this selector will get no data and no error.

Fix: Implement createThreadSelectorAcrossEnvironments to look up the thread by ID in state.environments[envId].threadShellById across all environments. Return the first match, or null if not found. Add a unit test asserting the selector returns a thread when one exists.

Risk: Very low. The fix is additive; null-returning stub is currently harmless.

Low F7 — resolveModelForThread and resolveThreadDetail do repeated SQLite reads per event

Files: apps/server/src/orchestration/Layers/ProviderRuntimeIngestion.ts:682–697

Problem: resolveModelForThread calls providerService.listSessions() (in-memory scan) and resolveThreadDetail calls projectionSnapshotQuery.getThreadDetailById (SQLite read) on every invocation. These are called on turn.completed and can also be called multiple times within a single complex event via the lazy-loaded getLoadedThreadDetail() closure. While the closure memoizes within one event, two events processed back-to-back for the same thread will each re-read from SQLite.

Why it matters: At very high event rates (fast streaming turns), this adds SQLite round-trips that accumulate. Lower priority than F1/F2 because the serial worker already limits concurrency and SQLite reads are fast for small tables.

Fix: Add a short-lived (50–100ms) thread-shell cache in ProviderRuntimeIngestion keyed by threadId, separate from the existing Effect caches. The thread shell (status, session, active turn) changes infrequently relative to streaming deltas. Invalidate on any lifecycle event (turn.started, turn.completed, session.exited).

Risk: Low-medium. Must invalidate the cache correctly to avoid stale reads on lifecycle events. Scope carefully.

Recommendation

Do these now, in order:

  1. F2 (WAL mode) — One or two lines, zero risk, immediate throughput improvement for all SQLite workloads. Do this first.
  2. F3 (LIMIT on list queries) — Defensive hygiene before usage stats features ship. Low effort, high safety value.
  3. F5 (Zustand selector memoization) — Low effort, visible rendering improvement during active turns, no server risk.
  4. F6 (storeSelectors stub) — One-shot fix while the module is fresh.
  5. F1 (Map-based read model) — Higher effort, higher reward. Plan as a standalone PR with projector unit tests.
  6. F4 (cache-key index) — Schedule after F1; depends on understanding the cache structure.
  7. F7 (thread-shell micro-cache) — Defer until profiling confirms it is actually a bottleneck.

Do not touch yet: The provider adapter layer (OpenCode, Anthropic), the checkpoint reactor, or the authentication flow — no performance issues were found there and the risk of regression is high.

Acceptance Criteria

  1. PRAGMA journal_mode returns wal when queried on the running server SQLite database.
  2. listByThreadId and listByProjectId accept a limit option and default to at most 200 rows.
  3. selectSidebarThreadsAcrossEnvironments is only computed once per render cycle when multiple overview selectors are used together.
  4. createThreadSelectorAcrossEnvironments("some-thread-id") returns the matching thread shell when that thread exists in state.
  5. pnpm typecheck passes with no new errors.
  6. Existing tests in apps/server/src and apps/web/src pass without modification.

Risks and Notes

  • WAL mode: WAL requires the database file and WAL file to be on the same volume. Always true here but worth documenting. If the server is run on a read-only filesystem (unlikely, but possible in some CI setups), WAL will fail to activate. Guard with a warning log rather than a crash.
  • LIMIT on list queries: Any existing usage statistics UI that calls the list queries expecting complete history will now silently truncate. Audit all consumers of listByThreadId/listByProjectId before the change lands. The aggregate queries are unaffected.
  • Selector memoization: React hook memoization (useMemo) only works inside components. Pure selector functions called outside React must use a library like reselect. Verify that workspaceOverviewSelectors.ts functions are only used inside React components before applying useMemo.
  • Read-model Map refactor (F1 — deferred): When this is eventually executed, the projection replay bootstrap must correctly rebuild the Map from persisted events. Snapshot serialization/deserialization must convert Map ↔ Array. This is the riskiest change and should have a dedicated PR with 100% projector coverage before merging.

Recommended Execution Sequence

Task 1 (low-risk, do now): WAL mode + LIMIT on list queries + storeSelectors stub fix

Task 2 (medium-risk, do next): Zustand selector memoization

Task 3 (high-risk, plan separately): Map-based read model (F1)

Defer: Cache-key index (F4), thread-shell micro-cache (F7)
