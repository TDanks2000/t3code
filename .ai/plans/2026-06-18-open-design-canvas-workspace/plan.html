Open Design Canvas Workspace Plan

TL;DR

Revise the feature direction: do not make this feel like a T3 Code chat with a preview side panel. Build a dedicated Open Design / Claude Design-style workspace route with a left context rail, full-grid canvas, top file/share bar, and floating bottom-left prompt composer. The existing chat/thread/provider engine should power it, but the UI should look and behave like a design canvas.

The first implementation should create a design workspace shell that can launch agent runs, bind context inputs, show generated artifacts as canvas frames, and keep the familiar Open Design loop: context → brief → artifact → critique → update/export.

1. Goal

Build a T3 Code-native version of Open Design that visually and behaviorally tracks the screenshot: a calm canvas-first UI where users start by adding context, then describe what they want to create, then watch an artifact appear on the grid. The workspace should feel closer to Claude Design/Open Design than to the existing T3 Code chat interface.

2. Codebase Understanding

  • T3 Code already has the hard part: provider sessions, thread state, composer draft state, file links, model selection, attachments, terminal/runtime mode, and WebSocket orchestration.
  • The current primary chat UI lives in apps/web/src/components/ChatView.tsx. This should be reused as behavior reference, not as the visual layout for this feature.
  • Routing is TanStack Router under apps/web/src/routes. This feature should likely become a dedicated route such as /design, /design/$designId, or a project-scoped design route.
  • The composer implementation in apps/web/src/components/chat/ChatComposer.tsx has reusable dispatch ideas but is visually too chat-oriented. Extract behavior or create design-specific components instead of forcing the existing footer into the screenshot layout.
  • apps/web/src/components/ui/* includes buttons, dialogs, menus, sheets, select, input, textarea, and tooltip primitives that can implement the screenshot without new UI dependencies.
  • apps/web/src/composerDraftStore.ts and apps/web/src/store.ts are relevant for draft/thread continuity.
  • Open Design's public model centers on context sources, design systems, prompt skills, sandboxed iframe previews, and exportable HTML artifacts. This plan adapts that model to T3 Code's existing local provider/session engine.

3. Recommendation

Replace the earlier "right preview panel" plan with a dedicated design workspace route. A side panel would not satisfy the requested Open Design / Claude Design feel. Build a separate canvas surface that internally uses T3 Code threads.

Do this in two execution slices. Slice 1 builds the shell, composer, context buttons, prompt construction, artifact frame, and generated HTML preview. Slice 2 adds richer Open Design parity such as selectable templates/skills, multi-artifact canvas layout, version history, and file-level critique tools.

4. Scope

In Scope

  • A dedicated Open Design-style route or mode.
  • Left sidebar with title row and "Start with context" actions: Design system, Screenshot, Figma.
  • Bottom-left floating composer with attach button, optional voice/tool button, model selector, and Send button.
  • Large right canvas with subtle grid background, top file selector, Share button, and account avatar.
  • Artifact frame rendering generated HTML in a sandboxed iframe on the canvas.
  • Prompt construction that tells the agent to generate/update a real HTML artifact under a stable project path.
  • State model for open file, selected artifact, context inputs, and generation state.
  • Focused tests for pure state/prompt/artifact logic.

Out of Scope For Slice 1

  • Full Open Design plugin marketplace.
  • Full Figma OAuth/API integration. The Figma button can open a paste/import dialog or disabled setup state first.
  • Video, HyperFrames, PPTX/PDF export, image model routing, or hosted BYOK proxy.
  • Multi-user sharing backend. The Share button can start as Copy Link / Export actions unless sharing already exists.
  • Replacing the existing T3 Code chat UI.

6. Implementation Plan

  1. Create design workspace state and helpers. Add a module such as apps/web/src/designWorkspace/designWorkspaceState.ts with pure types and functions for workspace title, context sources, prompt draft, artifact metadata, selected file, and generation state.
  2. Add prompt construction for Open Design parity. Add buildDesignWorkspacePrompt. It should include selected context, design-system text, screenshot notes, Figma link/import notes, target artifact path, and a contract requiring a single-file HTML output plus a short completion summary.
  3. Add a dedicated route. Create a route like apps/web/src/routes/design.tsx or project-scoped equivalent. The route should render the canvas UI directly, not embed ChatView.
  4. Build DesignWorkspaceShell. Implement the two-column shell with fixed left rail and flexible canvas. Match the screenshot proportions: about 300-316px left rail, rounded large canvas container, warm off-white surfaces, subtle grid background, and compact top bar.
  5. Build DesignContextRail. Include the title row, centered "Start with context" block, three pill actions, and bottom prompt composer. Context buttons should open dialogs or setup states. Screenshot should support image attachment if existing composer attachment code can be reused safely.
  6. Build DesignPromptComposer. Make it a compact card pinned to the bottom of the left rail. Include textarea, attach button, tool/voice placeholder button, model selector, and Send button. Reuse model selection logic where possible, but avoid pulling in the full chat footer UI.
  7. Build DesignCanvasArea. Add top file selector, Share button, avatar, grid canvas, empty state, generating state, and artifact frame state. Empty canvas should stay mostly blank like the screenshot.
  8. Build DesignArtifactFrame. Render generated HTML in a sandboxed iframe. Add small frame toolbar actions: Preview, Export HTML, Open in editor. Keep artifact chrome minimal so the canvas remains visually dominant.
  9. Connect to existing agent dispatch. Use the current project/thread/provider mechanisms from ChatView as reference. Prefer creating or reusing a thread behind the design workspace so runs still appear in existing session history. The design route can hide chat messages by default and only surface artifact/progress state.
  10. Implement artifact discovery and reading. Rank generated HTML files from the latest turn diff and explicit prompt target path. Read the file safely from the active project/worktree. Add a minimal typed read endpoint only if no existing project file read path exists.
  11. Add context dialogs. Design system: textarea or file picker for DESIGN.md-style context. Screenshot: image attachment. Figma: URL/token placeholder dialog with clear disabled/setup messaging if full import is not implemented in Slice 1.
  12. Add tests. Cover prompt construction, artifact candidate ranking, context source state transitions, and target path validation. Add component smoke tests for the shell if the existing web test setup supports it.

7. Code Quality Rules

  • Do not force the existing chat layout to look like Open Design. Build a dedicated design workspace surface.
  • Reuse provider/session behavior, not chat visuals.
  • Keep generated HTML sandboxed in an iframe.
  • Extract pure logic for prompt construction, path validation, and artifact discovery.
  • No any, no unsafe casts, and no schema weakening.
  • Use existing UI primitives and lucide icons.
  • Keep the palette warm, restrained, and close to the screenshot, while still using project tokens where practical.
  • Do not add external UI libraries.

8. Testing and Validation

  • Run vp check.
  • Run vp run typecheck.
  • Run focused web tests for changed design workspace helpers/components.
  • Run vp test if focused test selection is unclear.
  • Manual QA desktop at 1920x1080: the layout should visually match the supplied screenshot.
  • Manual QA laptop width around 1280px: left rail remains usable and canvas does not overflow horizontally.
  • Manual QA mobile/narrow width: stack or adapt the rail/canvas without overlapping text.
  • Manual QA generation loop: context → prompt → run → artifact appears → refine → artifact updates.

9. Acceptance Criteria

  • The user can open a dedicated design workspace that looks like the screenshot: left context rail, bottom-left composer, top file/share bar, and full grid canvas.
  • The first empty state says "Start with context" and offers Design system, Screenshot, and Figma actions.
  • The composer accepts a design brief, model selection is visible, and Send starts an agent-backed generation run.
  • The generated artifact appears on the canvas as a framed sandboxed HTML preview.
  • The top file selector reflects the open artifact or "No file open".
  • The feature preserves T3 Code's existing provider/session reliability rather than creating a parallel untracked execution path.
  • vp check and vp run typecheck pass.

10. Risks and Notes

Main risk: copying only the visual shell without using T3 Code's existing session engine would create a fragile parallel product. The implementation must keep execution rooted in the current provider/thread infrastructure.

  • Security: generated HTML is untrusted. Keep sandboxing strict and never inject it directly into app DOM.
  • Scope: "basically identical to Open Design" is too large for one slice. Build visual parity and the core artifact loop first.
  • Figma: real Figma import likely needs auth and API handling. Start with URL/context capture unless credentials and backend patterns already exist.
  • Navigation: decide whether design workspaces are per project, per thread, or standalone. Prefer project-scoped for artifact file safety.
  • Persistence: design workspace state should survive route changes and reloads enough to avoid losing prompts/context.
