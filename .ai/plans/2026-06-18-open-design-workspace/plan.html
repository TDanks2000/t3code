Open Design Workspace Plan

TL;DR

Build a focused T3 Code version of Open Design as a design-artifact mode inside the existing chat workspace, not as a separate app. The first slice should add a composer-level Design button, a right-side Design Preview panel with a sandboxed iframe, prompt templates for design artifacts, and export/download actions for generated HTML.

Do not attempt to clone Open Design's full desktop daemon, plugin marketplace, video pipeline, model router, or 100+ skill registry in the first pass. Use T3 Code's existing provider/session model, composer drafts, provider skills, right panel, and project files.

1. Goal

Create an agent-native design workspace inside T3 Code where the user can ask Codex, Claude, or another configured provider to generate real design artifacts. The user should click a visible Design control in the existing composer, choose an artifact type and design-system hint, send the brief through the current thread, and preview the latest generated HTML artifact in a sandboxed right panel.

The desired mental model is: brief → design mode → agent writes artifact → preview → critique/refine → export.

2. Codebase Understanding

  • Stack: React 19, Vite Plus, Tailwind CSS v4, TanStack Router, Zustand, Effect schemas, TypeScript 6, pnpm workspaces.
  • Required checks from AGENTS.md: vp check and vp run typecheck must pass before completion.
  • The chat surface is centralized in apps/web/src/components/ChatView.tsx. It renders ChatHeader, MessagesTimeline, ChatComposer, PlanSidebar, RightPanelSheet, terminal drawers, and image dialogs.
  • The composer already has the right insertion point in apps/web/src/components/chat/ChatComposer.tsx: model picker, provider traits, runtime mode, Plan/Build toggle, plan-sidebar toggle, and send actions live in the footer.
  • The right-panel pattern already exists through apps/web/src/components/RightPanelSheet.tsx, apps/web/src/rightPanelLayout.ts, and PlanSidebar.
  • Markdown rendering is sanitized in apps/web/src/components/ChatMarkdown.tsx. Do not put unsandboxed generated HTML into markdown. Preview generated artifacts through an iframe with a restrictive sandbox policy.
  • Open Design's public docs describe the relevant concepts: local-first design skills, DESIGN.md systems, sandboxed iframe preview, HTML/PDF/PPTX export, and artifact categories including prototypes, dashboards, decks, images, and video. Sources: nexu-io/open-design and open-design.ai.

3. Recommendation

Do this now as a smaller vertical slice. T3 Code already controls local provider sessions and has enough UI primitives to make design-artifact workflows useful without importing Open Design's architecture wholesale.

The first implementation should be UI/client-heavy and mostly additive. It should generate better prompts and preview generated HTML files from the current project/worktree. Persistence can start in the existing composer draft store or a small client runtime helper; server-side artifact indexing should wait until the preview flow is proven.

4. Scope

In Scope

  • Add a Design toggle/button beside existing composer mode controls.
  • Add a Design panel on the right using the same inline/sheet behavior as the Plan sidebar.
  • Add a Design Brief popover or compact drawer section for artifact type, design-system hint, target file path, and export intent.
  • Inject a structured prompt prefix/suffix when Design mode sends a message.
  • Detect and preview generated HTML artifacts from explicit file paths or from assistant output conventions.
  • Render the preview in an iframe with restrictive sandboxing and clear loading/empty/error states.
  • Add download/copy/open-in-editor controls for the active artifact.
  • Add focused unit tests for prompt construction, artifact detection, and UI state reducers/helpers.

Out of Scope

  • No plugin marketplace, skill registry clone, MCP server, desktop daemon, video rendering, PDF/PPTX export, external model router, or remote BYOK proxy in the first slice.
  • No changes to provider execution contracts unless previewing current-project files proves impossible.
  • No unsandboxed generated HTML injection into React markdown or DOM.
  • No new dependency unless the executor proves an existing browser/platform API is insufficient.

6. Implementation Plan

  1. Create client-only design workflow types. Add a small helper module such as apps/web/src/designWorkspace.ts or apps/web/src/lib/designWorkspace.ts. Define DesignArtifactType, DesignDraft, DesignArtifactCandidate, defaults, and pure helper functions. Keep runtime logic out of packages/contracts unless a server method is required.
  2. Build prompt composition as pure logic. Add buildDesignPrompt(input) that wraps the user's prompt with artifact type, design system, target path, preview contract, and safety rules. The prompt must tell the provider to write a real single-file HTML artifact under an explicit project-relative path and to mention that path in the final response.
  3. Add artifact detection helpers. Parse assistant text and turn diff summaries for likely project-relative .html files under paths such as design-artifacts/, artifacts/, or a user-selected target. Prefer files changed in the latest turn over older mentions.
  4. Add DesignPreviewPanel. Implement a panel component matching PlanSidebar and DiffPanel density. Include header, artifact controls, empty/loading/error states, iframe preview, and footer actions. The iframe must use a restrictive sandbox and should not use dangerouslySetInnerHTML.
  5. Wire panel state into ChatView. Add designMode, designPanelOpen, designDraft, and activeDesignArtifact state close to the existing Plan sidebar state. Use the same media query and RightPanelSheet pattern as PlanSidebar.
  6. Add the composer Design button. Put the primary button in ComposerFooterModeControls and the compact equivalent in CompactComposerControlsMenu. Use PaletteIcon or SparklesIcon from lucide-react, an accessible label, and an active state matching the Plan toggle treatment.
  7. Intercept send composition, not transport. When designMode is active, have ChatView transform the outgoing prompt using buildDesignPrompt before the existing send path dispatches it. Preserve images, terminal contexts, model selection, runtime mode, approvals, and existing error handling.
  8. Load artifact contents safely. First try existing local/project file APIs. If no typed read method exists, add a minimal schema-backed file-read contract near the existing project file APIs, with path normalization and project-root confinement on the server. Do not expose arbitrary absolute path reads.
  9. Add export/open actions. Implement Copy Path, Open in Editor, and Download HTML. Download can use a Blob from the loaded HTML; Open in Editor should reuse openInPreferredEditor or existing local API patterns.
  10. Add tests around logic and state. Test prompt construction, artifact candidate ranking, target path validation, compact control visibility, and send prompt transformation. Avoid brittle visual snapshots.
  11. Manual QA the full loop. Generate a simple landing page artifact, confirm the file is written, confirm the panel opens, confirm the iframe shows the result, refine the artifact in a follow-up prompt, and confirm export/open actions work.

7. Code Quality Rules

  • Follow existing component density and Tailwind token usage.
  • Keep the first slice additive and focused; do not restructure provider sessions.
  • Extract shared pure logic instead of embedding parsing and prompt construction inside React components.
  • Do not use any or weaken schema validation.
  • Do not duplicate Plan sidebar or Diff panel layout code when a small shared panel shell is warranted.
  • Do not introduce a design marketplace abstraction until there are at least two real local design systems or templates to manage.
  • Use icons from lucide-react and existing UI primitives.
  • Make async actions show pending state on the control that triggered them.

8. Testing and Validation

  • Run vp check.
  • Run vp run typecheck.
  • Run focused tests for changed web helpers/components, for example vp run --filter @t3tools/web test -- designWorkspace if the test naming supports it.
  • Run vp test if the focused test path is unclear.
  • Manually verify desktop and narrow widths. The Design panel must be inline on wide layouts and a sheet on narrow layouts.
  • Manually verify that generated HTML is only rendered in the sandboxed iframe.

9. Acceptance Criteria

  • A visible Design button exists in the composer footer, including compact/mobile controls.
  • Clicking Design opens or activates a right-side Design Preview panel without losing the current draft prompt.
  • Design mode sends a structured prompt that asks the provider to write a project-local single-file HTML artifact.
  • After an agent writes the artifact, the UI identifies the latest relevant HTML file and previews it in a sandboxed iframe.
  • The panel has empty, loading, error, and loaded states.
  • The user can copy the artifact path, open it in the preferred editor when available, and download the HTML.
  • The implementation passes vp check and vp run typecheck.

10. Risks and Notes

Security: Generated HTML is untrusted. Use iframe sandboxing and never merge it into React markdown or the app DOM.

  • Path safety: Any file-read API must confine reads to the active project or worktree. Normalize and reject traversal.
  • Provider variance: Different providers may not reliably write the requested artifact path. The prompt and candidate detector should handle both explicit path mentions and latest turn diffs.
  • Performance: Large HTML artifacts should not be reparsed every render. Cache artifact content by thread, path, and latest diff/update marker.
  • UX conflict: Plan sidebar and Design panel are both right-side panels. The first slice should allow one primary right panel at a time or make their toggles mutually understandable.
  • Scope creep: Open Design includes many product areas. Avoid marketplace, templates catalog, image/video generation, and automation until the preview loop is genuinely useful.
