Browser MCP Agent Capabilities

T3 Code · 2026-06-25 · Preview automation MCP toolkit

TL;DR

The existing MCP browser toolkit covers 12 operations but is missing four agent-blocking gaps: hover (no way to reveal tooltips or CSS :hover menus without evaluate), native <select> option picking, double-click / right-click (needed for context menus and rich editors), and multi-tab management (the agent cannot list, switch, or close tabs). This plan adds 6 new MCP tools and extends 1 existing one across 6 files — all additive, no breaking changes.

Codebase Understanding

Stack

  • MCP layer: Effect-TS 4.0 beta Tool / Toolkit from effect/unstable/ai, served over HTTP/SSE in apps/server/src/mcp/McpHttpServer.ts
  • Broker: apps/server/src/mcp/PreviewAutomationBroker.ts — a SynchronizedRef-backed queue that routes MCP requests to the connected Electron renderer owner
  • Renderer (owner): apps/web/src/components/preview/PreviewAutomationOwner.tsx — a React hook that picks up requests from the broker queue and dispatches them via a switch(request.operation) to either in-process state or window.desktopBridge.preview.automation.* IPC calls
  • Desktop IPC handlers: apps/desktop/src/ipc/methods/preview.ts — thin wrappers over PreviewManager
  • PreviewManager: apps/desktop/src/preview/Manager.ts (2,811 lines) — the main-process Chromium controller using CDP / Input.dispatchMouseEvent / Playwright injected runtime
  • Contracts: packages/contracts/src/previewAutomation.ts — all shared schemas and error classes

Dispatch chain for every operation

  1. MCP handler calls broker.invoke({ scope, operation, input })
  2. Broker enqueues the PreviewAutomationRequest to the matching owner's queue
  3. PreviewAutomationOwner.tsx handleRequest() receives it, matches request.operation in a switch
  4. For desktop-native operations: calls previewBridge.automation.* which IPC-invokes apps/desktop/src/ipc/methods/preview.ts
  5. Desktop IPC handler calls PreviewManager.automation*(tabId, input)
  6. Manager uses CDP (Input.dispatchMouseEvent, Runtime.callFunctionOn, etc.) to drive Chromium

Identified gaps

  • Hover: No Input.dispatchMouseEvent({ type: "mouseMoved" }) path. Agents must use preview_evaluate to manually dispatch mouseenter / mouseover events, which doesn't trigger CSS :hover pseudo-classes reliably.
  • Select elements: preview_type calls Input.insertText which does nothing on <select> elements. No other tool can pick an option.
  • Double-click / right-click: performAutomationClick hard-codes button: "left" and clickCount: 1 in its CDP calls. Context menus and double-click-activated editors are unreachable.
  • Multi-tab: The broker always resolves to owner.tabId (last focused tab). Agents cannot enumerate tabs, switch, or close one without user intervention.

Key constants and limits in Manager.ts

  • AGENT_CURSOR_MOVE_MS = 160 — delay between cursor move and click for animation
  • AGENT_CURSOR_CLICK_LEAD_MS = 40 — pre-click delay
  • MAX_INTERACTIVE_ELEMENTS = 200
  • MAX_VISIBLE_TEXT_LENGTH = 20_000
  • MAX_SCREENSHOT_WIDTH = 1280

Recommendation

Do this now. All six changes are additive — new operations in the literal union, new tools in the toolkit, new cases in the switch. Nothing is removed or renamed. The risk of breaking existing behaviour is low. The value is high: hover and select unblock entire categories of form and navigation interactions that currently require fragile preview_evaluate workarounds, and multi-tab support unlocks authentication flows that require opening a login page in a second tab.

Scope

In Scope

  1. preview_hover — new tool: move mouse to element, trigger CSS :hover state, hold for tooltip dwell time
  2. preview_select — new tool: pick a <select> option by value, visible label, or zero-based index
  3. preview_click extension — add optional button ("left" / "right" / "middle", default "left") and clickCount (1 / 2, default 1) to the existing click schema; no new tool needed
  4. preview_tab_list — new tool: return all open tabs for the current thread with their URL, title, loading state, and which one is active
  5. preview_tab_switch — new tool: make a specific tabId the active tab
  6. preview_tab_close — new tool: close a specific tab

Out of Scope

  • File upload / drag-and-drop (not confirmed as a pain point)
  • Iframe / shadow DOM targeting (complex, low frequency)
  • Network interception / mocking
  • Snapshot quality improvements (accessibility tree pruning, Playwright-style selector generation) — separate follow-up
  • Human-facing browser panel UI changes (tab bar, navigation chrome)

Code Quality Rules

  • Do not use any. All new schemas must use Effect Schema types.
  • Reuse resolveClickPoint for hover's point resolution — do not duplicate the locator-resolution logic.
  • Reuse focusAutomationTarget logic pattern for select's element resolution — or extract a shared resolveElement helper if the duplication would be more than 20 lines.
  • Follow the Effect.fn("PreviewManager.automationHover") span-naming convention.
  • All new input schemas must match the annotation style of existing ones: .annotate({ description: "..." }) on every field.
  • New channel constants must follow the PREVIEW_AUTOMATION_*_CHANNEL naming in IpcChannels.
  • No new npm dependencies. The Playwright injected runtime, CDP, and Effect are sufficient.
  • The PreviewAutomationOperation literal union is checked exhaustively in TypeScript switch statements. The compiler will flag any missed cases — fix them all.

Risks and Notes

  • tabList renderer-side data shape: The tabList case reads from previewStateStore in the renderer. The exact field names (sessions, activeTabId, navStatus) must be verified against apps/web/src/previewStateStore.ts before implementation. A wrong field name will silently return empty arrays.
  • tabSwitch and broker owner: After switching tabs, the broker's invoke resolves the owner's tabId from the PreviewAutomationOwner state. The reportCurrentAutomationOwner() call in the tabSwitch case must fire after the store is updated, not before.
  • hover dwell time: 300ms is sufficient for most tooltips. Very slow tooltip animations (e.g., 500ms+ CSS transitions) will require a higher dwellMs. Document this in the tool description.
  • double-click CDP dispatch: Some browsers fire dblclick only when two mousedown events arrive within the system double-click interval. With CDP, sending two press+release pairs in sequence with clickCount: 2 on both is the correct pattern — but the CDP clickCount field is specifically for this. Test against a dblclick event listener to confirm.
  • select by label: The substring match in the evaluate JS is case-sensitive. If the page trims whitespace from option text, the match may fail for labels with leading/trailing spaces. Trim both sides in the comparison.
  • Toolkit exhaustiveness: Both PreviewToolkit and PreviewStandardToolkit in tools.ts must include the new tools. Forgetting to add them to one of the two toolkits will cause them to silently not appear in that MCP session type.
